Effective Java 读书笔记 （一）

标签： 读书

Stay hungry，stay foolish. 继续啃书。

第一条：考虑用静态工厂方法代替构造器

静态工厂方法与构造器不同的第一大优势在于，它有名称。如果构造器的参数本身没有确切的描述正被返回的对象，那么具有适当名称的静态工厂会更容易使用。产生的客户端代码也更容易阅读。例如，构造器BigInteger（int,int,Random）返回的BigInteger可能为素数，如果用名为BigInteger.probablePrime的静态工厂方法来表示，显然更为清楚。（1.4版本增加了这个方法）。

静态工厂方法与构造器不同的第二大优势在于，不必再每次调用它的时候都创建一个新对象。这时得不可变类可以使用预先构建好的实例，或者将构建好的实例缓存起来，进行重复利用，从而避免创建不必要的重复对象，如果程序经常请求创建相同的对象，并且创建对象的代价很高则这项技术可以极大的提升性能。
静态工厂方法能够为重复的调用返回相同的对象，这样有助于类总能严格控制在某个时刻那些实例应该存在。这种类被称作实例受控的类。编写实例受控类有几个原因。实例受控类可以确保他是一个Singleton或者是不可实例化的。

静态工厂方法与构造器不同的第三大优势在于，它们可以返回原返回类型的任何子类对象。这样我们在选择返回对象是可以更灵活。

静态工厂的第四大有死在于，在创建参数实例化的时候，它们是代码更加简洁。

静态工厂方法的缺点在于，类如果没public或者protected的构造器，就不能被子类化

静态工厂方法的第二个缺点在于，它们与其他的静态方法实际没区别。简而言之，静态工厂通常更加合适，因此第一反应就是提供构造器，而不先考虑静态工厂。

第二条遇到多个构造器参数时要考虑用构建器

静态工厂和构造器都有一个共同的局限性：它们都不能很好的扩展到大量的参数。下面就是一个Builder的实例：

public class NutritionFacts{
    private final int servingSize;
    private final int servings;
    private final int calories;
    private final int fat;
    private final int sodium;
    private final int carbohydrate;

    public static class Builder{
        //Required parameters
        private int servingSize
        private int servings;

        //Optional parameters - initialized to default values
        private final int calories      = 0;
        private final int fat           = 0;
        private final int sodium        = 0;
        private final int carbohydrate  = 0;

        public Builder(int servingSize, int servings){
            this.servingSize = servingSize;
            this.servings = servings;
        }

        public Builder calories(int val){
            calories = val;
            return this;
        }
        public Builder fat(int val){
            fat = val;
            return this;
        }
        public Builder sodium(int val){
            sodium = val;
            return this;
        }
        public Builder carbohydrate(int val){
            carbohydrate = val;
        }

        public NutritionFacts build(){
            return new NutritionFacts(this);
        }
    }

    private NutritionFacts(Builder builder){
        servingSize = builder.servingSize;
        servings = builder.servings;
        calories builder.calories;
        fat = builder.fat;
        sodium = builder.sodium;
        carbohydrate = builder.carbohydrate;
    }
}

注意NutritionFacts是不可变的，所有的默认参数都单独放在一个地方。builder的setting方法返回builder本身，以便可以把调用链接在起来。下面是客户端代码：

NutritionFacts cocaCola = new NutritionFacts.Builder(240,8).calories(100).sodium(35).carbohydrate(27).builde();

这样的客户端代码很容易编写，更为重要的是，易于阅读。builder模式模拟了具名的可选参数，就像Ada和Python中的一样。

第三条用私有构造器或者枚举类型强化Singleton属性

Singleton即单例模式，在Java1.5发行版之前，实现Singleton有两种方式。这两种方式都需要把构造器私有化，并到处静态成员，以便允许客户端能够访问该类的唯一实例。
第一种方式：

//Singleton with public final field
public class Elvis{
    public static final Elvis INSTANCE = new Elvis();
    private Elvis(){}
    public void leaveTheBuilding(){...}
}

由于构造器是私有的，所以能保证Elvis实例的唯一性。Tip:可以借助AccessibleObject.setAccessible()方法，通过放射机制调用构造器。如果需要抵御这种攻击，可以修改构造器，让它在创建第二个实例时抛出异常。
第二种方式：

//Singleton with static factory
public class Elvis{
    private static final Elvis INSTANCE = new Elvis();
    private ELvis(){}
    public static Elvis getInstance(){
        return INSTANCE;
    }
    public void leaveTheBuilding(){...}
}

对于静态方法Elvis.getInstance(),都会返回同一个实例对象。

工厂方法的优势之一在于，它提供了灵活性：在不改变API的前提下，我们可以改变该类是否为Singleton。

从Java1.5发行版起，实现Singleton还有第三种方法。只需编写一个包含单个元素的枚举类型：

//Enum Singleton - the preferred approach
public enmu Elvis{
    INSATANCE;
    public void leaveTheBuilding(){...}
}

这种方法功能上与共有域方法相近，但是它更简洁，无偿的提供序列化机制，绝对防止多次实例化，即使在面对复杂的序列化或反射攻击的时候。虽然这种方法还没有广泛的使用，但是单元素的枚举方法已成为实现Singleton的最佳方法。
